# Phase 2: Checkout Hardening Before Payment

Phase 2 prepares checkout for a real payment gateway without locking the application to a provider yet.

## Order Placement Safety

- The storefront order placement flow now runs product reads with row locks before totals are calculated.
- Stock deduction is guarded by a conditional update, so a product can only be decremented when enough stock remains at the moment of update.
- The cart is cleared only after order rows, vendor rows, payment placeholders, stock updates, and loyalty updates commit successfully.

## Duplicate Submission Protection

- Checkout requests carry a `Checkout_Request_Key` generated by the storefront and sent in the `Idempotency-Key` header and request body.
- `Orders_Placed_T.Checkout_Request_Key` has a unique index so retries cannot create duplicate orders.
- If the same checkout request is submitted again, the API returns the existing order response instead of creating a new order.

## Shipping Quote Validation

- The cart page stores `quoted_at` and `expires_at` with the selected shipping option.
- The order API rejects expired quotes and recalculates the current quote before accepting the selected shipping option.
- If the selected shipper, destination, basis, or price no longer matches, the order is rejected and the customer must refresh shipping options.

## Payment Abstraction

- Payment statuses are gateway independent: `unpaid`, `pending`, `paid`, `failed`, `refunded`, `partially_refunded`.
- `App\Services\Checkout\PaymentGateway` is bound to `PendingPaymentGateway` until a real gateway is selected.
- Payment placeholders are stored on the sales transaction details table with gateway, intent, idempotency, and metadata fields.
